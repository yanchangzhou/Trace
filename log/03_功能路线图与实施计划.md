# Trace 功能路线图与实施计划

生成日期：2026-05-09  
目标：把当前项目从“本地文档编辑 + AI 侧栏”推进到“个人写作风格生成工作台”。

## 北极星指标

用户上传 3-10 篇自己的历史作品后，能在 10 分钟内生成一篇指定文体的草稿，并认为它“明显比通用 AI 更像自己、更容易继续改”。

如果这个指标达不到，其他功能都不是核心。

## 产品主链路

```text
创建项目/Book
  -> 上传资料和写作样本
  -> 标记哪些文件用于知识资料，哪些文件用于风格样本
  -> 系统解析、清洗、切分
  -> 生成风格画像
  -> 用户编辑/确认风格画像
  -> 选择文体模板
  -> 输入写作目标
  -> AI 生成草稿
  -> 插入编辑器
  -> 用户修改/采纳/反馈
  -> 反馈写回风格画像和生成记录
```

## 阶段 0：先稳定当前工程

时间建议：2-4 天

### 目标

保证当前应用在 Tauri 环境下可以稳定启动、上传、预览、保存 note、调用 AI。

### 任务

   明确只以 `npm run tauri dev` 作为核心能力验收方式。浏览器模式只用于 UI 预览。

2. 补基础 README

   - 安装依赖
   - 开发启动
   - Tauri 启动
   - API key 设置
   - 支持文件格式

3. 梳理当前命令  
   给所有 Tauri commands 建一张表：命令名、输入、输出、调用方、是否已稳定。

4. 增加最小测试  
   至少覆盖：
   - `parser::split_into_chunks`
   - `style::extract_style_profile` 的空数据和基础数据
   - DB migration 能跑通

### 验收标准

- 新机器按 README 能启动。

- note 自动保存可用。
- AI panel 能完成一次流式生成。

## 阶段 1：文档解析和样本库

时间建议：1-2 周

### 目标

让用户上传的文件真正变成可分析的文本资产，并区分“知识资料”和“风格样本”。

### 后端任务

1. 新增文件用途字段  
   在 `files` 表增加：

```sql
ALTER TABLE files ADD COLUMN role TEXT NOT NULL DEFAULT 'source';
-- source: 知识资料
-- style_sample: 风格样本
-- both: 同时作为资料和风格样本
```

2. 新增解析状态字段

```sql
ALTER TABLE files ADD COLUMN parse_status TEXT NOT NULL DEFAULT 'pending';
ALTER TABLE files ADD COLUMN parse_error TEXT;
```

3. 建立 `document_pipeline`

```text
copy file
  -> create file record(status=importing)
  -> extract text
  -> clean text
  -> extract structure
  -> split chunks
  -> write documents/chunks
  -> index chunks
  -> update status=ready
```

4. 修 PDF 文本提取  
   这是 P0。不要继续用 `String::from_utf8_lossy(&bytes)`。

5. 结构化 chunk  
   chunk 至少包含：
   - `file_id`
   - `chunk_index`
   - `text`
   - `heading_path`
   - `start_char`
   - `end_char`
   - `page` 或 `slide`

### 前端任务

1. 文件列表支持标记用途  
   在 SourceRail 中为每个文件提供：
   - 用作资料
   - 用作风格样本
   - 同时使用

2. 显示解析状态  
   不要让用户猜文件能不能用于 AI。显示 pending、parsing、ready、failed。

3. 失败可重试  
   解析失败的文件提供 retry。

### 验收标准

- 上传 PDF/DOCX/TXT/MD 后可看到真实抽取文本。
- 用户能把文件标成风格样本。
- 解析失败有明确错误，不会静默 ready。

## 阶段 2：风格画像 MVP

时间建议：1-2 周

### 目标

从“统计几个数字”升级为“可用于生成的风格画像”。

### 风格画像 JSON 建议

```json
{
  "profile_name": "我的公众号风格",
  "language": "zh-CN",
  "sample_summary": {
    "file_count": 8,
    "word_count": 18500,
    "date_range": null
  },
  "tone": {
    "formality": "semi-formal",
    "emotion": "restrained but opinionated",
    "confidence": "assertive"
  },
  "structure": {
    "opening_patterns": ["从具体场景切入", "先提出反常识判断"],
    "paragraph_style": "短段落为主，每段 2-4 句",
    "ending_patterns": ["回到开头问题", "给出行动建议"]
  },
  "sentence": {
    "length": "短中句为主，偶尔长句解释逻辑",
    "patterns": ["先判断，再解释", "反问后转折"]
  },
  "vocabulary": {
    "preferred_words": [],
    "connectors": [],
    "avoid_words": []
  },
  "examples": [
    {
      "text": "原文片段",
      "why_it_matters": "体现了开头方式"
    }
  ],
  "generation_rules": [
    "不要过度营销",
    "不要使用夸张标题党",
    "每个核心观点后给一个具体例子"
  ]
}
```

### 后端任务

1. 新增 `style_profiles` 和 `style_examples` 表。
2. 增加命令：
   - `create_style_profile_from_samples`
   - `list_style_profiles`
   - `get_style_profile_detail`
   - `update_style_profile`
   - `delete_style_profile`
3. 风格提取分两层：
   - 本地统计：句长、段落、标点、词频、标题结构。
   - LLM 总结：把统计和样本片段整理成可读画像。
4. 每个画像必须保存原文 examples，不能只有抽象描述。

### 前端任务

1. 新增风格画像页面或面板。
2. 用户可以查看并编辑风格规则。
3. 用户可以删除不合适的样本片段。
4. 用户可以创建多个画像：
   - 我的公众号风格
   - 我的正式邮件风格
   - 我的学术写作风格

### 验收标准

- 选择 3 个 TXT/DOCX 样本能生成一个画像。
- 画像不是空泛描述，必须带原文片段。
- 用户能编辑画像并保存。
- AI 生成时实际使用该画像。

## 阶段 3：文体模板和生成工作流

时间建议：1-2 周

### 目标

把 AI 从“聊天”变成“可交付写作任务”。

### 首批只做三个模板

1. 微信公众号推文
2. 长邮件
3. 课程论文/作业辅助

不要一开始做太多。模板越多，质量越差。

### WritingTask 类型建议

```ts
interface WritingTask {
  task_type: 'wechat_article' | 'long_email' | 'course_paper';
  style_profile_id?: string;
  source_file_ids: string[];
  prompt: string;
  audience?: string;
  goal?: string;
  length?: string;
  language?: 'zh-CN' | 'en-US' | 'bilingual';
  constraints?: string[];
  output_mode: 'outline' | 'draft' | 'rewrite' | 'polish';
}
```

### 后端任务

1. 新增 `prompt_builder.rs`  
   按 task type 组织 prompt，不要继续在 `main.rs` 里拼字符串。

2. 新增 `generation_runs` 表

```sql
CREATE TABLE generation_runs (
  id TEXT PRIMARY KEY,
  task_type TEXT NOT NULL,
  style_profile_id TEXT,
  source_file_ids_json TEXT NOT NULL DEFAULT '[]',
  model TEXT NOT NULL,
  prompt_json TEXT NOT NULL,
  output_text TEXT NOT NULL DEFAULT '',
  status TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);
```

3. 检索增强  
   根据用户 prompt 对 source files 做相关 chunk 检索，不要只取前 10 个 chunks。

4. 输出引用  
   如果生成内容基于资料，返回引用映射。

### 前端任务

1. AI 面板增加任务选择器。
2. 每个任务有专门输入表单：
   - 公众号：主题、目标读者、观点、长度、是否标题党。
   - 长邮件：收件人、关系、目的、语气、必须包含的信息。
   - 课程论文：题目、课程要求、引用资料、字数、论点。
3. 生成结果支持：
   - 插入编辑器
   - 替换选区
   - 复制
   - 重新生成
   - 生成更像我的版本

### 验收标准

- 用户可以选一个风格画像和一个文体模板生成草稿。
- 草稿能一键插入编辑器。
- 生成记录被保存。
- 生成时能看到用了哪些来源文件。

## 阶段 4：编辑器内 AI

时间建议：1 周

### 目标

让 AI 成为写作动作，而不是旁边的聊天窗口。

### 功能

1. 选区改写
   - 更正式
   - 更口语
   - 更像当前风格画像
   - 降低 AI 味
   - 缩短
   - 扩写

2. Slash 命令
   - `/公众号`
   - `/长邮件`
   - `/论文提纲`
   - `/引用资料`
   - `/改成我的风格`

3. 版本对比
   - 原文
   - AI 改写
   - 用户采纳

4. 反馈按钮
   - 像我
   - 不像我
   - 太正式
   - 太啰嗦
   - 太营销

### 验收标准

- 用户不打开 AI 侧栏，也能通过选区和 Slash 完成常见操作。
- 每次采纳/拒绝都能记录到 generation feedback。

## 阶段 5：风格评估与反馈学习

时间建议：1-2 周

### 目标

让系统知道“生成得像不像”，并可持续改进。

### 评估方式

1. 规则指标
   - 句长分布差异
   - 段落长度差异
   - 标点使用差异
   - 高频词覆盖
   - 标题/小标题结构

2. LLM judge
   - 输入样本片段、风格画像、生成结果。
   - 输出哪里像、哪里不像、如何修改。

3. 用户反馈
   - 采纳率
   - 手动修改距离
   - 点赞/点踩原因

### 新增表

```sql
CREATE TABLE generation_feedback (
  id TEXT PRIMARY KEY,
  generation_run_id TEXT NOT NULL,
  rating TEXT,
  reason TEXT,
  edited_text TEXT,
  created_at INTEGER NOT NULL
);
```

### 验收标准

- 每篇生成结果有风格匹配报告。
- 用户反馈会影响下一次生成 prompt。
- 能看到“为什么不像”的具体解释。

## 验收 Demo 脚本

1. 用户创建一个 Book：`我的公众号写作`。
2. 上传 5 篇自己写过的文章，标记为 `style_sample`。
3. 上传 2 份参考资料，标记为 `source`。
4. 点击“生成风格画像”。
5. 系统展示风格画像和原文片段。
6. 用户编辑一条规则：`不要使用过度营销语气`。
7. 用户选择“微信公众号推文”模板。
8. 输入主题：`为什么年轻人越来越需要个人知识库`。
9. 系统生成标题、大纲和完整草稿。
10. 用户点击“插入编辑器”。
11. 用户选中第二段，点击“更像我的风格”。
12. 系统给出修改，并显示“与当前风格画像匹配度 78%”。

如果这个 demo 跑顺，Trace 才算进入真正产品阶段。

## 最后提醒

现在最危险的不是代码没写完，而是主线不够锋利。Trace 应该先证明自己能做一件难而有价值的事：让 AI 写出来的东西更像用户本人。

这个目标没有打穿之前，所有“更多格式、更漂亮 UI、更多模板、更多模型”都只是外围。
