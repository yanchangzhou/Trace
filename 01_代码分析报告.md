# Trace 代码分析报告

生成日期：2026-05-09  
分析范围：`/Users/chowyc/Documents/Trace` 当前工作区代码  
产品目标：让用户上传自己的文本/文档，分析写作风格，并按目标场景生成微信公众号推文、长邮件、课程论文作业等文案。

## 一句话结论

现在的 Trace 已经有一个像样的本地知识写作软件雏形：Next.js + Tauri 桌面端、文件导入、SQLite 数据库、Tantivy 搜索、Tiptap 编辑器、AI 面板和流式生成都存在。

但它还不是“个人写作风格生成器”。目前更接近“本地文档管理 + 编辑器 + 一个接了 OpenAI 的侧边聊天框”。真正决定产品成败的三件事还很弱：可靠解析用户样本、结构化风格画像、按画像稳定生成并可评估改写。

## 当前完成情况

### 已经做得不错的部分

1. 桌面应用路线是对的  
   Tauri 适合处理用户本地文件、隐私数据、离线索引和本地数据库。对“上传个人写作样本”这个场景来说，桌面端比纯 Web 更容易建立信任。

2. 数据层已经有雏形  
   `src-tauri/src/db.rs` 已经有 `books`、`files`、`documents`、`document_chunks`、`notes`、`note_sources`、`note_versions`、`settings` 等表。这个方向正确，说明你已经从纯前端 demo 走向真正应用。

3. 文档导入和索引链路已经开始打通  
   `copy_file_to_book` 会复制文件、异步解析、写入 DB、切 chunk、写入搜索索引。虽然粗糙，但这是关键主链。

4. 编辑器已经不只是 textarea  
   `components/editor/EditorShell.tsx` 使用 Tiptap，支持自动保存、标题、富文本、引用插入、Slash 命令和基础工具栏。这比很多 AI 写作 demo 更接近真实写作工作台。

5. AI 流式输出已经具备  
   `src-tauri/src/ai.rs` 里已经实现 OpenAI Chat Completions 流式请求，前端 `components/ai/AIPanel.tsx` 通过 Tauri event 接收 token。AI 能“跑起来”。

### 还没有完成的核心能力

1. “上传文件并分析其写作风格”没有真正实现  
   现在的 `style.rs` 只从用户 notes 中统计平均句长、高频词、标题密度、词汇丰富度、段落长度和非常粗糙的 tone。它不是从用户上传样本里学习风格，也不能输出稳定的风格模板。

2. “以用户需要的风格为模板生成文案”只实现了皮毛  
   `my_style` 在 `ai.rs` 中只是加了一句 `Write in the user's personal style based on their writing history.`。这句话对模型几乎没有约束力。模型不知道用户的句式节奏、常用转折、结构偏好、语气边界、禁用词、开头结尾习惯，也不知道不同文体该如何迁移。

3. 文件内容解析不可靠，尤其是 PDF  
   `parser.rs` 的 PDF 解析只是读取 bytes；`parse_to_text` 对 PDF 使用 `String::from_utf8_lossy(&bytes)`，这对大多数 PDF 是无效文本。也就是说，PDF 样本越多，系统越可能拿乱码做风格分析和检索。

4. AI 生成没有真正进入编辑器工作流  
   AI 面板可以生成文本，但生成结果主要留在聊天区；没有“插入到编辑器当前位置”“按选区改写”“生成多个版本”“采纳/拒绝修改”“保留引用来源”的完整写作体验。

5. 没有风格评估闭环  
   生成后没有判断“像不像用户”。缺少 style similarity score、样本对比、人工反馈、偏好记忆和版本迭代。没有评估，就很难调 prompt 和模型策略。

## 关键代码问题

### 1. `main.rs` 过大，后端命令层开始失控

`src-tauri/src/main.rs` 约 609 行，承担了太多职责：Tauri 启动、路径管理、文件选择、文件复制、数据库 CRUD、索引、解析、AI、版本历史、style profile、watcher 启动。

典型问题：

- `copy_file_to_book` 同时负责文件系统复制、重名处理、异步索引、解析、DB 写入和搜索索引写入。
- `stream_generate` 同时负责读取 API key、构造 prompt、调 context、spawn AI 请求。
- 业务分层不清晰，后续加“风格样本库”“文体模板”“生成任务队列”时会继续膨胀。

建议拆分：

- `src-tauri/src/commands/files.rs`
- `src-tauri/src/commands/books.rs`
- `src-tauri/src/commands/notes.rs`
- `src-tauri/src/commands/ai.rs`
- `src-tauri/src/services/document_pipeline.rs`
- `src-tauri/src/services/style_profile.rs`
- `src-tauri/src/services/generation.rs`

不要继续把命令堆进 `main.rs`。现在还能忍，再加两轮功能就会变成不能维护的总线文件。

### 2. 文档解析能力和产品目标不匹配

产品目标依赖高质量文本提取，但当前解析很浅。

问题位置：

- `src-tauri/src/parser.rs:61-79`：PDF preview 只读 bytes，metadata word_count 是 0。
- `src-tauri/src/parser.rs:336-341`：PDF 文本提取把二进制 bytes 当 UTF-8 文本读，这是错误方向。
- `src-tauri/src/parser.rs:347-367`：DOCX 只抽段落 run text，不处理表格、脚注、标题样式、列表结构、批注、页眉页脚。
- `src-tauri/src/parser.rs:370-390`：PPTX 只读 slide XML 字符内容，不保留文本框结构、层级、备注、顺序和版式语义。
- `src-tauri/src/parser.rs:424-467`：chunk 只按字符数和简单标点切分，不理解标题、段落、章节和语义边界。

对产品的影响：

- 用户上传课程论文、公众号文章 PDF、邮件导出、PPT 课件时，风格样本可能解析不完整。
- 检索内容不稳定，AI context 会缺失关键内容。
- 生成时所谓“参考用户风格”没有可靠样本基础。

建议优先级：

1. PDF 必须接入真实文本提取。Rust 侧可评估 `pdf-extract`、`lopdf`、`pdfium-render`，或用 sidecar 调 `pdftotext`。如果要跨平台打包，优先验证 Tauri bundle 后的可用性。
2. DOCX 解析要保留段落类型、heading level、list、table。不要只要纯文本。
3. PPTX 至少保留 slide number、shape order、speaker notes。
4. chunk 应按结构切：heading -> section -> paragraph -> semantic chunk，而不是固定 2000 字符。

### 3. “风格分析”目前只是统计，不是风格画像

`src-tauri/src/style.rs` 当前实现的问题比较明确。

问题位置：

- `src-tauri/src/style.rs:16`：先调用 `db.list_notes_by_book("")`，注释也承认“不太可能工作”。这行没有实际意义，甚至可能引入错误。
- `src-tauri/src/style.rs:21-31`：只遍历 notes，不分析上传的源文件。产品目标是用户上传文件分析风格，这里入口错了。
- `src-tauri/src/style.rs:44-179`：平均句长、高频词、标题密度、词汇丰富度、段落长度、被动语态启发式，最多只能算“文本统计”，不是可用于生成的风格模型。
- `src-tauri/src/ai.rs:59-66`：`my_style` 只是固定一句 prompt，没有把 `StyleProfile.constraints` 注入生成请求。

真正需要的风格画像至少应包括：

- 结构：常见开头方式、段落推进方式、标题层级、结尾方式。
- 句式：长短句比例、排比、反问、转折、括号、破折号、冒号使用习惯。
- 语气：正式/口语、克制/煽动、确定/保留、主观/客观。
- 词汇：高频表达、禁用词、偏好词、行业术语、连接词。
- 文体迁移：同一个人的公众号风格和课程论文风格不应混为一谈。
- 示例库：每个风格结论都应该绑定 3-5 个原文片段，生成 prompt 才有抓手。

建议新增表：

```sql
CREATE TABLE style_profiles (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  source_scope TEXT NOT NULL,
  language TEXT,
  profile_json TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);

CREATE TABLE style_examples (
  id TEXT PRIMARY KEY,
  profile_id TEXT NOT NULL,
  file_id TEXT,
  note_id TEXT,
  text TEXT NOT NULL,
  tags_json TEXT NOT NULL DEFAULT '[]',
  FOREIGN KEY(profile_id) REFERENCES style_profiles(id) ON DELETE CASCADE
);
```

### 4. AI 生成链路太薄，不能支撑多场景写作

当前 AI action 只有：

- summarize
- compare
- outline
- free chat

见 `components/ai/AIPanel.tsx:51-56`。

这和产品目标中的“微信公众号推文、长邮件、课程论文作业等内容”还差很远。不同文体不是 prompt 里换一个词这么简单。至少要有：

- 写作场景模板：公众号、长邮件、课程论文、课程作业、商业 proposal、社媒短文。
- 任务阶段：生成大纲、扩写、改写、润色、降低 AI 味、压缩、标题生成、摘要、续写。
- 输入结构：目标读者、目的、语气、字数、必须包含的信息、禁止出现的信息、引用文件范围。
- 输出结构：标题、摘要、正文、引用、可复制版本、可编辑版本。

后端 `src-tauri/src/main.rs:395-408` 目前只根据 action 拼一个 `action_prompt` 和 `style_instruction`，没有任务模板系统，也没有 schema 化输出。

建议：

1. 新增 `WritingTask` 类型，替代现在过于简单的 `AIRequest`。
2. 每种文体维护 prompt 模板和输出 schema。
3. 生成时使用三段式 prompt：
   - `System`: 产品规则、隐私规则、引用规则。
   - `Style`: 结构化风格画像 + 原文示例。
   - `Task`: 用户本次目标、素材、格式要求。
4. 生成完成后写入 `generation_runs` 表，保留 prompt、模型、输入来源、输出、用户采纳情况，方便以后优化。

### 5. AI context 构造粗糙，容易丢内容、塞错内容

`src-tauri/src/ai.rs:10-56` 的 `build_ai_context` 直接取每个文件前 10 个 chunks，每个 chunk 截 500 字符。这会导致：

- 用户问文档后半部分，context 可能完全没有。
- 多文件时无排序、无召回、无相关性判断。
- 没有引用 locator，AI 生成后不能回溯来源。
- 可能把不相关内容塞进 prompt，浪费 token。

建议：

- 用户有明确 prompt 时，先对 prompt 做 `search_content`，取 top-k chunk。
- 用户选择文件时，先按章节摘要 + 相关 chunk 混合构造 context。
- 每个 chunk 必须带 `file_id`、`file_name`、`chunk_id`、`locator_json`。
- 生成回答时强制输出引用映射，前端显示来源。

### 6. 搜索索引和 DB 状态可能不一致

`copy_file_to_book` 会先复制文件，然后异步解析和索引。函数立即返回路径，前端随后 `reindexFiles` 和 `refreshLibrary`。这里有竞态：

- 异步任务还没写入 DB，前端刷新可能看不到文件。
- `reindexFiles` 只索引文件名/metadata，不等于解析 chunk 完成。
- 解析失败时，UI 仍可能显示 ready。
- 没有 indexing、parsing、failed、ready 的明确状态机。

建议：

- 上传时先写 `files(status='importing')`。
- 解析成功后更新为 `ready`。
- 失败更新为 `failed` 并记录 `error_message`。
- 前端轮询或监听 Tauri event 更新状态。
- 不要让前端猜测“上传完成 = 可用于 AI”。

### 7. API key 明文存 SQLite，不适合真实用户

`src-tauri/src/db.rs:523-536` 把 `openai_api_key` 存在 settings 表。对早期开发可以接受，但对真实产品不合格。

建议：

- macOS 用 Keychain，Windows 用 Credential Manager，Linux 用 Secret Service。
- Tauri 可接入 secure storage 插件或自封装。
- UI 上明确告诉用户 key 存在哪里、是否会上传、如何删除。

### 8. 前端组件过大，状态和 UI 耦合

比较明显的大文件：

- `components/ai/AIPanel.tsx` 约 530 行。
- `components/SourceRail.tsx` 约 488 行。
- `components/BookSelector.tsx` 约 486 行。
- `lib/tauri.ts` 约 448 行。

问题不是“行数大就是错”，而是这些文件同时混合了 UI、状态、业务流程、Tauri 调用、浏览器 fallback、错误处理。后续加文体模板、风格选择、引用来源、生成历史时，会继续变臃肿。

建议拆分：

- `components/ai/AIPanel.tsx` 只保留布局和组合。
- `components/ai/StyleSelector.tsx`
- `components/ai/TaskSelector.tsx`
- `components/ai/PromptInput.tsx`
- `components/ai/MessageList.tsx`
- `hooks/useAIStreaming.ts`
- `hooks/useStyleProfile.ts`
- `lib/tauri/books.ts`
- `lib/tauri/files.ts`
- `lib/tauri/ai.ts`

### 9. 浏览器 fallback 会稀释产品判断

`BookContext` 和 `SourceRail` 都有非 Tauri 浏览器模式，用 localStorage 和 object URL 模拟文件库。开发阶段可以理解，但你的目标产品明显是本地文件 + 桌面端。

问题：

- 同一套 UI 里混了桌面真实逻辑和浏览器假逻辑。
- 用户在浏览器模式上传的文件不会进入真正解析/索引/AI 链路。
- 很容易在测试时误以为功能完成。

建议：

- 如果主产品定位桌面端，就把 browser mode 限定为 UI demo，并在代码和界面中明确标识。
- 核心能力验收只认 Tauri 环境。
- 不要继续为 browser fallback 投入复杂逻辑，除非你决定做 Web SaaS。

## AI 写作功能是否已经实现

结论：部分实现了“调用模型生成文本”，没有实现“以用户风格为模板的写作产品”。

已实现：

- 保存 OpenAI API key。
- Tauri 后端调用 OpenAI Chat Completions。
- 流式 token 传回前端。
- 用户可选择 default、academic、analytical、concise、my_style。
- 可以选择 summarize、compare、outline、chat。

未实现或很弱：

- 上传样本后自动建立 style profile。
- 根据不同文体建立不同风格子画像。
- 生成公众号、邮件、论文等结构化文案。
- 根据引用资料生成并标注来源。
- 插入编辑器、选区改写、版本对比。
- 生成结果的风格相似度评估。
- 用户反馈驱动的风格迭代。
- 模型供应商抽象和中国大陆可用模型支持。

尖锐一点说：现在的 AI 功能能证明“API 接通了”，不能证明“产品有壁垒”。

## 建议借鉴的产品

### NotebookLM：借鉴“来源驱动”

NotebookLM 的核心不是“聊天”，而是让用户围绕 sources 建立 notebook，并基于来源回答、总结、生成内容。Trace 应该学习它的来源感：

- 每次生成都明确用了哪些文件和哪些片段。
- 回答旁边显示引用来源。
- 支持对某个文件/某组文件提问。
- 支持自动生成摘要、FAQ、briefing、study guide 一类派生材料。

但不要照抄 NotebookLM 的定位。NotebookLM 更偏“理解资料”，Trace 应该偏“用我的风格写出可交付文本”。

### Notion：借鉴“编辑器内 AI”

Notion AI 的优势在于 AI 嵌入文档编辑工作流，而不是单独聊天窗口。Trace 应该把 AI 变成编辑器动作：

- 选中一段 -> 改成我的风格。
- 输入 `/公众号` -> 生成公众号结构。
- 输入 `/长邮件` -> 生成邮件草稿。
- 选中文章 -> 降低 AI 味。
- 选中文章 -> 更像某个 style profile。

现在 Trace 的 AI 侧栏太像“旁边站着的聊天机器人”，还没有进入写作手部动作。

### Jasper / Grammarly：借鉴“品牌声音/写作规范”

Jasper 的 Brand Voice 和 Grammarly 的 tone/clarity 类能力说明：用户愿意为“一致的声音”和“稳定的质量控制”付费。

Trace 可以做自己的版本：

- 我的正式邮件风格。
- 我的公众号风格。
- 我的课程论文风格。
- 我的中文小红书风格。
- 我的英文申请文书风格。

每个风格都应该可命名、可编辑、可测试、可比较，而不是一个笼统的 `my_style`。

## 功能建议

### MVP 必须补齐

1. 样本库  
   用户可以把文件标记为“风格样本”，而不是所有资料都混在一起。

2. 风格画像生成  
   对样本输出结构化 JSON：语气、结构、句式、词汇、禁忌、示例。

3. 文体模板  
   公众号、长邮件、课程论文至少三类先做深，不要一开始覆盖十几类。

4. 编辑器内生成  
   支持插入当前位置、替换选区、生成多个候选版本。

5. 引用与溯源  
   如果生成基于文件资料，必须能回到源片段。

6. 结果评估  
   给出“风格匹配度”和问题提示，例如：句子太长、开头不像、语气过于营销。

### 值得加但不急的功能

- 风格 A/B 对比。
- 个人语料清洗：去掉引用、目录、参考文献、他人文字。
- 风格漂移提醒：最近写作和历史风格差异。
- 多语言风格迁移：中文样本生成英文邮件，英文样本生成中文推文。
- 团队风格库：公司公众号风格、老板邮件风格、课程老师偏好。
- 本地模型模式：只做离线风格统计和小模型润色。
- 模型路由：OpenAI、Claude、Gemini、DeepSeek、通义千问、智谱等。

## 优先修改建议

### P0：先把“风格样本 -> 风格画像 -> 生成”的主链跑通

不要继续优先做 UI 美化或更多文件预览。产品验证最需要的是：

1. 上传 3 篇用户写过的文章。
2. 点击“生成我的公众号风格画像”。
3. 输入主题。
4. 生成一篇接近用户风格的推文。
5. 用户可以指出哪里不像，系统记住。

只要这条链路跑通，产品才真正开始。

### P1：修 PDF 和 chunk

PDF 解析不修，很多真实样本不可用。chunk 不修，RAG 和风格提取都会飘。

### P1：把 prompt builder 独立出来

不要在 `main.rs` 里继续拼 prompt。后续文体模板、风格画像、引用规则都会让 prompt 快速复杂化。

### P2：拆前端大组件

现在拆还不晚。等 AI 面板继续堆功能，会非常难拆。

### P2：API key 安全存储

真实用户的 key 不能长期明文放 SQLite。

## 最后判断

这份代码不是“烂”，它的问题是方向已经从 demo 进入产品，但代码结构还停在 demo 阶段。继续加功能会越来越慢，因为主链路、模块边界和数据模型还没有为“个人风格写作”这个核心目标服务。

最该停下来的不是开发，而是“无目标地加功能”。下一步要非常残酷地围绕一个问题推进：用户上传自己的作品后，Trace 能不能生成一段让用户觉得“这确实像我会写出来的东西”？

如果答案还是不能，那 NotebookLM、Notion、Grammarly、Jasper 都不是主要敌人。主要敌人是产品自己还没有证明核心价值。
